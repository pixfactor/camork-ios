## Problem

Build 19 fixed several real-device issues, but the Gallery top chrome still looked wrong on device. The custom header/fade path produced an opaque gray slab over content, fought the large-title interaction, and made the list feel less native than the map surface.

## Decision

Build 20 removes the custom Gallery top chrome and returns the top edge to the system NavigationStack large title behavior. The list mode picker now lives in normal scroll flow, and the bottom tab bar uses system material configuration instead of an app-specific fade mask.

This is closer to the Codex mobile app style the user asked for: system-owned top chrome, content that can visually sit under a translucent bottom control, and no hand-drawn header band.

  - restored `.navigationTitle("gallery_title")`
  - restored `.navigationBarTitleDisplayMode(.large)`
  - moved the list/map picker into normal content flow
  - removed the custom top/bottom gallery scroll-edge overlay path from the list screen
- `Camork/RootTabView.swift`

  - kept the tab bar visible with `.toolbarBackground(.bar, for: .tabBar)`

  - regenerated by `xcodegen generate`

## Rejected

- Custom gray/material header overlay: rejected because it looked like a stuck modal sheet on real device and obscured the large-title transition.
- Mask-based top/bottom fades: rejected because they crop card shadows and image edges, making the UI look clipped instead of polished.
- More aggressive full-bleed list under a handmade header: rejected because it recreates the same z-order problem and is fragile across real-device safe areas.

## Remaining Risk

The final authority for this change is real-device visual testing through TestFlight build 20. The simulator confirmed the structural fix, but the user still needs to install build 20 and verify the top/bottom feel during live scrolling.
